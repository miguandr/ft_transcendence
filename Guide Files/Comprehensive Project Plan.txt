Key Design Decisions
	1	Monolithic first, modular structure: Single backend app with clear module separation, easier to deploy/debug for students
	2	REST + WebSockets hybrid: REST for CRUD, WebSockets layered on top for real-time
	3	PostgreSQL with ORM: Relational data (orgs → projects → sprints → tasks) fits well
	4	JWT authentication: Stateless, works well with WebSockets
	5	Server-side analytics computation: Simple queries, no heavy client processing
	6	Email notifications as enhancement: Core notifications in-app, email as bonus

3. Database Schema Outline
Core Tables
-- USERS & AUTH
Users
├── id (UUID, PK)
├── email (unique)
├── password_hash
├── name
├── avatar_url (nullable)
├── created_at
└── updated_at

-- ORGANIZATIONS
Organizations
├── id (UUID, PK)
├── name
├── owner_id (FK → Users.id)
├── created_at
└── updated_at

OrganizationMembers
├── id (UUID, PK)
├── organization_id (FK → Organizations.id)
├── user_id (FK → Users.id)
├── role (ENUM: 'admin', 'member')
├── joined_at
└── UNIQUE(organization_id, user_id)

-- PROJECTS & SPRINTS
Projects
├── id (UUID, PK)
├── organization_id (FK → Organizations.id)
├── name
├── description (nullable)
├── created_at
└── updated_at

Sprints
├── id (UUID, PK)
├── project_id (FK → Projects.id)
├── name
├── start_date
├── end_date
├── goal (nullable, text)
├── status (ENUM: 'planning', 'active', 'completed')
├── created_at
└── updated_at

-- TASKS (Sprint Board)
Tasks
├── id (UUID, PK)
├── sprint_id (FK → Sprints.id)
├── title
├── description (nullable, text)
├── status (ENUM: 'todo', 'doing', 'done')
├── assignee_id (FK → Users.id, nullable)
├── priority (ENUM: 'low', 'medium', 'high')
├── due_date (nullable)
├── created_by (FK → Users.id)
├── created_at
└── updated_at

-- ASYNC STANDUPS
StandupEntries
├── id (UUID, PK)
├── sprint_id (FK → Sprints.id)
├── user_id (FK → Users.id)
├── yesterday (text)
├── today (text)
├── blockers (text, nullable)
├── created_at
└── UNIQUE(sprint_id, user_id, DATE(created_at))

-- BLOCKER WORKFLOW
Blockers
├── id (UUID, PK)
├── sprint_id (FK → Sprints.id)
├── raised_by (FK → Users.id)
├── assigned_to (FK → Users.id, nullable)
├── title
├── description (text)
├── status (ENUM: 'open', 'in_progress', 'resolved')
├── resolved_at (nullable)
├── created_at
└── updated_at

-- COMMENTS (polymorphic)
Comments
├── id (UUID, PK)
├── entity_type (ENUM: 'standup', 'blocker', 'task')
├── entity_id (UUID)
├── user_id (FK → Users.id)
├── content (text)
├── created_at
└── updated_at

-- NOTIFICATIONS
Notifications
├── id (UUID, PK)
├── user_id (FK → Users.id)
├── type (ENUM: 'blocker_assigned', 'comment_added', 'task_assigned', etc.)
├── title
├── message
├── link (nullable, e.g., '/projects/123/sprint/456')
├── is_read (boolean, default false)
├── created_at
└── INDEX(user_id, is_read, created_at)

-- FILES
Files
├── id (UUID, PK)
├── uploader_id (FK → Users.id)
├── entity_type (ENUM: 'blocker', 'comment', 'task')
├── entity_id (UUID)
├── filename
├── original_name
├── mime_type
├── size_bytes
├── storage_path
├── created_at
└── INDEX(entity_type, entity_id)
Indexes & Performance
	•	Add indexes on foreign keys
	•	Composite index on (sprint_id, created_at) for standup queries
	•	Full-text search index on Tasks.title, Tasks.description for advanced search
